THE BASICS

The BAOBAB Places Plugin allows aggregation of locations; primarily through a longitude/latitude pair (in degrees), and, possibly, with other attributes, such as addresses, attached data items and even images or other media.

LOCATION OBFUSCATION

The BAOBAB Places Plugin has the ability to implement Location Obfuscation. It does this by applying a "fuzz factor" (discussed below) to a location's longitude/latitude pair.

The "fuzz factor" is a general distance from the actual location's long/lat. It is given in Kilometers.

Unless the logged-in user has the authorization to view the "raw" location, they will always get a different answer when querying that location. This will be a random location within a square around the actual location. It isn't 100% perfect, as a "brute force" attack could determine the location relatively accurately, but it is good enough for most tasks. The BAOBAB server does have the ability to improve the algorithm used for this.

In the above image, the red star represents the real location, and the purple circles are ones that are returned whenever the location is queried. Every query will result in a different location.

It is up to the data manager to ensure that other "hints" (like an address) are restricted or eschewed.

BASIC URI STRUCTURE

http[s]://<YOUR SERVER NAME>/[json|xml|xsd]/places

You access the plugin functionality by placing the plugin name immediately following the response type in the REST URI.

There is only one main directory -the main "places" plugin directory. Most functionality for this plugin comes from the query parameters.

Direct Place Record IDs

The one exception is that you can specify a list of one or more specific locations by their IDs, in a subdirectory-style CSV list, like so:

https://example.com/json/places/2,3,456,7890

If you specify a list of locations by ID, then you cannot search by radius from a location (covered below), or paginate the response (also covered below). Only the specified place records will be returned (assuming that the current user has at least read permissions on the records). If records to which the logged-in user does not have read permission are specified, they will be ignored.

If you call the Places Plugin with no search queries or ID subdirectories, the BAOBAB Server will return a list of all possible places, which could be a very large dataset.

QUERY PARAMETERS

"Global" Parameters

These are parameters that apply to all Places Plugin queries; whether a location search, or a list of IDs

show_details

This applies only to GET.

If you specify this (no value required), then the resulting list will display the places in "detailed" form:

writeable

This applies to GET, PUT and DELETE.

This only applies if you are logged into the BAOBAB server. If you specify this (no value required), then the resulting list will consist only of records to which your logged-in user has modification permissions. Note that PUT and DELETE require that the user have modification permissions on the record, so this could be an effective "prefilter," to avoid records that the user cannot affect:

https://example.com/[json|xml]/places?writeable https://example.com/[json|xml]/places/2,3,...?writeable

Search Parameters

These are parameters that apply to searches (no list of discrete IDs).

search_count_only

This applies only to GET.

If you specify this (no value necessary), then the response will not be a list of data items. Instead, you will get only a single integer, with the count of items that the specified search would return.

search_ids_only

This applies only to GET.

If you specify this (no value necessary), then the response will not be a list of full data items. Instead, you will get a list of integers, with each integer being the resource ID of the location in the response.

String Search Parameters

These apply to GET, PUT and POST.

There are a number of parameters that can be applied to perform a string search (not a long/lat or location radius search).

You can use SQL-style wildcards in these searches ("%" -percent sign).

These are case-insensitive.

search_name

This is the object name (not the venue name).

search_venue_name

This is the venue name.

search_street_address

This is the street address.

search_extra_information

This is the "extra information" column.

search_town

This is the city or town.

search_county

This is the county or sub-province.

search_state

This is the state or province.

search_postal_code

This is the ZIP code or postal code. It is a string (not an integer).

search_nation

This is the nation.

search_tag8
search_tag9

These are two "arbitrary string" values that can be used to store pretty much anything (up to 256 characters long).

Pagination

These queries apply to GET, PUT and DELETE.

You can get the search results back in "pages" of a specific number of items (or less) per page; specifying each of these "pages."

PUT and DELETE will operate on these results, so you can use pagination to determine which items will be processed. Be aware, however, that using DELETE changes the count and arrangement of records, so the operation could be unpredictable if repeated.

For example, a response of 1278 items, paged in groups of 20, would result in 64 pages (the last page would only have 19 items).

search_page_size=n

Where "n" is an integer of 1 or greater; dictating the maximum number of items per page.

Specifying this will result in the response being broken into a maximum of "n" results per page (which can be modified by the "show_details", "writeable", "search_ids_only" or "search_count_only" queries).

This works in concert with the "search_page_number=n" query (discussed below).

https://example.com/[json|xml]/places?search_page_size=20
search_page_number=n

Where "n" is an integer, with the 0-based index of the requested page.

The response data will begin from the first record that would appear on page n of the paged response, and will proceed fora maximum of "n" records, as specified by the "search_page_size=n" query.

This works in concert with the "search_page_size=n" query, discussed above. If the "search_page_size=n" is not provided (or set to 0), then this is ignored.

https://example.com/[json|xml]/places?search_page_number=2

If this is not specified, but "search_page_size=n" is, then the response will be the first page of results (the equivalent of "search_page_number=0").

Locality Searches

These queries apply to GET, PUT and DELETE.

It is possible to perform searches in the BAOBAB server for places near locations. These locations can be specified by a longitude/latitude pair, or, if the server is set up to allow it, an address that is looked up and resolved into a longitude/latitude pair.

You specify a search radius around the location, in Kilometers. The reults are returned with an element that describes the search location and radius, and each record is given a "straight line" distance from that point (also in Kilometers).

search_radius

This query is required for both of the following queries. It is the radius from the search center, in Kilometers (as a floating-point number). The search will be in a circle around the center point. "Fuzzy" locations may appear outside the circle.

https://example.com/[json|xml]/places?search_radius=10.5
Direct Longitude/Latitude Searches

Direct longitude/latitude searches do not require any special server configuration, and are applied immediately.

search_longitude

This is the longitude, in degrees, for the search center (as a floating-point number).

https://example.com/[json|xml]/places?search_longitude=-77.46723
search_latitude

This is the latitude, in degrees, for the search center (as a floating-point number).

https://example.com/[json|xml]/places?search_latitude=42.5678
Address Lookup Searches

The server can be set up to allow the REST client to supply a readable address, which is then looked up, and resolved (if possible) into a longitude/latitude pair.

This requires enabling in the server configuration, because it uses the server Google API key to do a geocode lookup. This is a service that could cost money, so the server administrator needs to control it.

It is possible to disable the feature, make it available only to logged-in users, or make it available to all vistors; whether or not they have a login. If it is disabled or restricted for the current visitor, the lookup (and the search) fails.

search_address

This is the address, as a readable string, for the search center.

https://example.com/[json|xml]/places?search_address=350+Fifth+Avenue,+New+York+City,+NY
search_region_bias

This is "hint" or Region Bias, to help Google determins where the address "fits" (For example, Toledo, Ohio vs. Toledo, Spain).

This is ignored if "search_address=xxx" is not used.

https://example.com/[json|xml]/places?search_region_bias=us
Response Data

If you specify a center/radius search, the response data will have an extra "search_location" element, with the search center and radius. Additionally, each record will also have a "distance_in_km" element, with a floating-point number. This will denote the distance, in Kilometers, from the element's long/lat to the search center. It should be noted that "Fuzzy" location distance will be inaccurate by design.

POST and PUT Parameters

DELETE simply uses the resource URI, and does not use any query arguments.

However, POST and PUT allow you to selectively set fields using URI queries. These queries apply to PUT and POST, and your logged-in user must have modification (write) permissions on the record(s) being modified:

name

This is simply the record name.

read_token

This is a new integer read token for the record. It must be one that is available to you.

write_token

This is a new integer write token for the record. It must be one that is available to you.

lang

This is the language code for the user and/or login. It should be the 2-letter abbreviation for the code.

owner_id

This is the integer ID of an "owner" record for this object. This is not a particularly useful field. It is mostly used for "triaging" large datasets.

longitude

This is the new longitude for the place. It is a floating-point number. Note that for "fuzzy" records, this will set the "raw" longitude, but checking it subsequently will give a different result. In order to check this, you should query for the "raw_longitude."

latitude

This is the new latitude for the place. It is a floating-point number. Note that for "fuzzy" records, this will set the "raw" latitude, but checking it subsequently will give a different result. In order to check this, you should query for the "raw_latitude."

fuzz_factor

This is how you set (or clear) the "fuzziness" of a location-obfuscated record. If you set this to 0, then the record will no longer be "fuzzy." Otherwise, set it to a floating-point number that will describe a square, twice the given number of Kilometers to a side, surrounding the true location. Subsequent "blind" queries will give random locations within this square.

can_see_through_the_fuzz

This is an integer (specifying this empty will clear the field in the record). This is an ID, other than the write permission ID, that can view the "raw_*" versions of the longitude and latitude of "fuzzy" records. This is a way to allow users to view the "real" location without giving the user full write permission to the place record.

address_venue

This is the "building name" of the place. It's usually something like "St. Mark's Episcopalian Church", or "Orchard Grove Middle School."

address_street_address

This is the street number/road address/PO Box of the place address, like "123 Main Street, Suite 1234," or "Pepe Street At LePew," or "PO Box 1234."

address_extra_information

This is "extra information" about the address. It can be things like directions, or descriptions of the venue ("Red Brick Building On the Northeast Corner").

address_town

This is the city/town of the address. It can also be a borough.

address_county

This is the county/sub-province of the address.

address_state

This is the state/province of the address.

address_postal_code

This is the postal/ZIP code for the address of the place.

address_nation

This is the nation for the place. It should be the two- or three-letter designation for the nation.

tag8

This is an arbitrary string to attach to an easily-searched part of the record.

tag9

This is another arbitrary string to attach to an easily-searched part of the record.

child_ids

Places are actually "collection" records. This means that they can have attached "child" records (of any type). You associate these records by providing their integer IDs as a CSV list in this query.

You specify IDs to be added by indicating them as positive integers, and to be removed, by specifying them as negative integers (prefixed by "-").

For example:

https://example.com/xml/places/1234?child_ids=134,2,-3,-4,5,674,-32

Would add IDs 134, 2, 5, and 674, and would remove IDs 3, 4, and 32 to the place with the ID of 1234 (if the record originally had them). If the record does not have the IDs, or the records indicated by the IDs are not visible to the current user, the IDs are ignored.

Order does not matter, but your login ID must have at least read permissions for the resource indicated by the child ID. If you do not have that, that child ID will be ignored; even if the record has it.

You can apply a PUT command to multiple records. Anything in the PUT URI that does not apply to a specific record is simply ignored for that record.

Uploading Binary Data
