BASIC URI STRUCTURE

Plugin Name

The plugin name is "people", all lowercase:

http[s]://<YOUR SERVER NAME>/[json|xml]/people

You access the plugin functionality by placing the plugin name immediately following the response type in the REST URI.

Main Resource Directories

There are two possible main resource directories:

People:

http[s]://<YOUR SERVER NAME>/[json|xml]/people/people

and Logins:

http[s]://<YOUR SERVER NAME>/[json|xml]/people/logins

The 'people' directory is for general data database records that contain information, such as surnames, personal data, addresses, photographs and other information.

The 'logins' directory is for security database records that pertain ONLY to the act of logging into the BAOBAB server. You cannot log into the BAOBAB server unless you have a login record.

In most cases, each login also has an associated 'people' record, but it is possible to have 'people' records with no login, and 'login' records with no associated 'people' record.

This plugin is the only way to directly access logins (security database items).

DIRECTORIES

The resources available to the user are determined by the login permissions of the user, and the directories and query arguments passed in.

Basic Directories

If you are logged in as a manager or server administrator, then you will have the ability to use POST, PUT and DELETE for most resources. If you are logged in as a regular user, then these methods will only work on your own user and login.

In all cases, only resources that your logged-in user is cleared to view or modify are made available.

User and Login Directories

You can access users and logins via their numeric (database) IDs, or their string (login) IDs. In the case of string logins, only users that have associated logins can be accessed.

Open Query (No IDs Specified)

If you call either of the main directories with no subdirectories, then it is assumed that you wish to view/modify all visible users or logins:

http[s]://<YOUR SERVER NAME>/[json|xml]/people/people http[s]://<YOUR SERVER NAME>/[json|xml]/people/logins

If you wish to view/modify specific user or logins, then you can specify them via one or more of their database integer IDs, as a CSV list:

http[s]://<YOUR SERVER NAME>/[json|xml]/people/people/n,n... http[s]://<YOUR SERVER NAME>/[json|xml]/people/logins/n,n...

Remember that logins usually have different IDs from their associated users.

You can also specify resources using login (string) IDs (Note that the subdirectory "login_ids" is appended to the "people/people" directory):

http[s]://<YOUR SERVER NAME>/[json|xml]/people/people/login_ids/LoginID1,LoginID2... http[s]://<YOUR SERVER NAME>/[json|xml]/people/logins/LoginID1,LoginID2...

There is a special directory that you can use to access your own login resources:

http[s]://<YOUR SERVER NAME>/[json|xml]/people/people/my_info http[s]://<YOUR SERVER NAME>/[json|xml]/people/logins/my_info

Specifying 'my_info' will result in only one record being returned (your own user or login).

QUERY ARGUMENTS

Resource Selection Arguments

These arguments apply to GET, PUT and DELETE.

writeable

This only applies if you are logged into the BAOBAB server. If you specify this (no value required), then the resulting list will consist only of records to which your logged-in user has modification permissions:

https://example.com/[json|xml]/people/people/<SUBDIRECTORY>?writeable https://example.com/[json|xml]/people/logins/<SUBDIRECTORY>?writeable
show_details

If you apply this to GET commands, the returned resources will be in 'expanded' format, with a great deal of information. This will not apply to the "my_info" subdirectory, which is already returned in 'expanded' format.

http[s]://<YOUR SERVER NAME>/[json|xml]/people/people/<SUBDIRECTORY>?show_details http[s]://<YOUR SERVER NAME>/[json|xml]/people/logins/<SUBDIRECTORY>?show_details

If you specify the "login_user" query parameter in a "people" subdirectoy POST call, then the plugin will .

POST AND PUT

In this plugin, POST does only one thing: It creates new users and logins; one at a time (or a matched pair at once). In order to POST, your login ID must be a "manager" ID. PUT is used to modify existing resources.

NOTE: Specifying just the people subdirectory, or just the logins subdirectory will cause just that resource to be created, with no matching second resource (a simple user record, with no login, or a simple login, with no user).

NOTE: Standalone users (only "people") have no ability to log into the BAOBAB server. Standalone logins, however, can log into the BAOBAB server.

Query Arguments (Logins)

login_string

If you are specifying a new login (specifying the logins subdirectory), or you are also attaching the login_user query parameter, you are required to provide a "login_string" parameter with the POST command. This should be a string, and will be the login ID of the new user. This is not allowed in the PUT method (you can't change a user ID), unless you are the "God" login.

password

If you are specifying a new login (specifying the logins subdirectory), or you are also attaching the login_user query parameter, you have the option of providing your own password in the "password" query parameter. This should be a string, and should be the minimum length (or greater) required by the BAOBAB installation. Leaving this out, or providing too short of a password, will result in a random password being generated (and returned in the request response).

name

You can optionally specify a name parameter. This is a string, and is put in the "object_name" record column in the new login.

tokens

If you are specifying a new login (specifying the logins subdirectory), or you are also attaching the login_user query parameter, you have the option of providing a CSV (Comma-Separated Values) list of integers, which will be security tokens for the new login. These tokens MUST be ones that your manager ID has available. You cannot assign any tokens that your own ID does not have.

is_manager

If you are specifying a new login (specifying the logins subdirectory), or you are also attaching the login_user query parameter, you have the option of specifying the "is_manager" query parameter. If this is specified, then the resulting login will be a "manager" login, with the ability to create other logins. Remember that this login will be restricted to whatever tokens you provide, as its own "pool" of tokens.

Query Arguments (People)

login_user

If the people subdirectory is specified, then you can assign the login_user query parameter. Doing this will cause the new user to be accompanied by a login ID, as well as a user. This parameter is ignored if you are specifying the logins subdirectory.

name

You can optionally specify a name parameter. This is a string, and is put in the "object_name" record column in the new user/login. If you are specifying a new login (specifying the login_user query argument), this will be applied to BOTH the user and the login.

surname

You can optionally specify a surname (family name). This applies only to the user record (not the login).

given_name

You can optionally specify a given (first) name. This applies only to the user record.

middle_name

You can optionally specify a middle name. This applies only to the user record.

prefix

You can optionally specify a prefix (like "Dr.", "Ms.", "Sir", etc.). This applies only to the user record.

suffix

You can optionally specify a suffix (like "esq.", "III", "LCSW", etc.). This applies only to the user record.

nickname

You can optionally specify a nickname. This applies only to the user record.

child_ids

Users (not logins) are actually "collection" records. This means that they can have attached "child" records (of any type). You associate these records by providing their integer IDs as a CSV list in this query.

You specify IDs to be added by indicating them as positive integers, and to be removed, by specifying them as negative integers (prefixed by "-").

For example:

http://example.com/xml/people/people/?login_user&child_ids=134,2,-3,-4,5,674,-32

Would add IDs 134, 2, 5, and 674, and would remove IDs 3, 4, and 32 (if the user record originally had them). If the user record does not have the IDs, or the records indicated by the IDs are not visible to the current user, the IDs are ignored.

Order does not matter, but your login ID must have at least read permissions for the resource indicated by the child ID. If you do not have that, that child ID will be ignored; even if the user has it.

You can apply a PUT command to multiple users and/or logins. Anything in the PUT URI that does not apply to a specific record is simply ignored for that record.

Uploading Binary Data

You can upload binary data (such as a user avatar image) with the POST or PUT command. The main description discusses the exact mechanism for doing this.
